# 智慧智能体 · 部署配置指南

## 概述

智慧智能体是一个对接**飞书项目（Meegle）**的 AI 分析助手。当用户在飞书 IM 中 `@智小协` 并发出分析指令时，自动从飞书项目拉取需求 PRD，调用 DeepSeek 完成分析，将报告以飞书文档形式返回给相关人。

### 三大核心能力

| 能力 | 触发方式 | 产出 |
|------|----------|------|
| 需求澄清 | 关键词 `分析` / `需求澄清` | 需求澄清问题清单（飞书文档） |
| 技术可行性初评 | 关键词 `分析` / `技术可行性` | 技术可行性初评报告（飞书文档） |
| PRD 超时提醒 | 定时轮询（每日 9:00） | 超时提醒消息 |

### 报告分发对象

| 角色 | 收到什么 |
|------|----------|
| 触发人 | 需求澄清 + 技术报告（发到聊天窗口） |
| 需求提出人 | 需求澄清 |
| 技术可行性确认节点负责人 | 技术报告 + 需求澄清 |

## 三、部署步骤

### 3.1 创建飞书应用（IM + 文档）

1. [飞书开放平台](https://open.feishu.cn) → 创建**自建应用**
2. 记录 **App ID**（`cli_...`）和 **App Secret**

**事件与回调**：
- 事件配置 → 使用长连接 → 添加 `im.message.receive_v1`
- 回调配置 → 使用长连接 → 添加 `card.action.trigger`

**权限管理**（全部添加并发布）：

| 权限 | 用途 |
|------|------|
| `im:message` | 发送消息 |
| `im:message.receive_v1` | 接收消息 |
| `im:message.p2p_msg:readonly` | 读取私聊 |
| `im:message.group_at_msg:readonly` | 读取群聊 |
| `im:chat:readonly` | 读取会话 |
| `im:resource` | 上传附件 |
| `wiki:wiki` | 读取 Wiki（OAuth 授权） |
| `wiki:node:read` | 读取 Wiki 节点 |
| `docx:document:readonly` | 读取文档内容 |
| `drive:drive` | 上传文件到云空间 |

**发布应用**（审核通过后权限生效）。

### 3.2 创建飞书项目插件

1. [飞书项目](https://project.feishu.cn) → 空间设置 → 开放平台 → 创建插件
2. 记录 **Plugin ID** 和 **Plugin Secret**
3. 获取 **User Key**：飞书项目页面 → 左下角双击头像自动复制

### 3.3 配置环境变量

```bash
cd 智慧智能体
cp config.env.example config.env
```

编辑 `config.env`，填入真实凭据：

```env
# 飞书项目平台
FEISHU_PROJECT_PLUGIN_ID=MII_xxx
FEISHU_PROJECT_PLUGIN_SECRET=xxx
FEISHU_PROJECT_USER_KEY=7649567855117192178
PROJECT_SPACE_KEY=aniwonder

# 飞书 IM + 文档
FEISHU_APP_ID=cli_xxx
FEISHU_APP_SECRET=xxx

# DeepSeek
DEEPSEEK_API_KEY=sk-xxx

# 运行配置
FEISHU_WS_ENABLED=true
```

### 3.4 配置管理员

编辑 `config/admins.json`，填入管理员的飞书 open_id：

```json
{
  "admins": ["ou_xxxxxxxxxxxxxxxxxxxx"]
}
```

> 管理员可通过 IM 对话新增/删除智能体。

### 3.5 安装依赖并启动

```bash
npm install
npm run dev     # tsx watch 热重载模式
```

启动成功输出：

```

启动后如果看到：

```

## 四、使用方式

### 4.1 飞书 IM 触发分析

在飞书中私聊 `智小协` 机器人，发送：

```
分析 萝卜蹲游戏
```

系统会：
1. 从飞书项目中搜索「萝卜蹲游戏」
2. 读取 PRD 文档（飞书 Wiki）
3. 调用 DeepSeek 做需求澄清 + 技术可行性分析
4. 生成两份飞书文档
5. 向触发人、提出人、技术负责人分发报告卡片

### 4.2 确认交互

如果需求名称不够精确，系统会列出候选：

```
找到「拍摄与智能影像」，是要分析这个需求吗？回复"是"继续。
```

回复 `是` / `对` / `确认` 即可继续。

### 4.3 HTTP 端点

| 端点 | 方法 | 用途 |
|------|------|------|
| `/health` | GET | 健康检查 |
| `/trigger/new-requirement/:id` | POST | 手动触发需求分析 |
| `/webhook/project` | POST | 飞书项目 Webhook |
| `/api/agents` | GET | 查看已注册智能体 |
| `/api/agents/register` | POST | 注册新智能体（管理员） |
| `/api/agents/unregister` | POST | 删除智能体（管理员） |
| `/auth/feishu-login` | GET | Wiki 读取授权 |
| `/auth/callback` | GET | OAuth 回调 |
| `/wiki/read` | GET | 读取 Wiki 文档内容 |

### 4.4 手动触发

```bash
curl -X POST "http://localhost:3456/trigger/new-requirement/7039179458?requester=ou_xxx&chat_id=oc_xxx"
```

## 五、数据流详解

```
飞书用户 "分析 萝卜蹲"
  │
  ▼
飞书 IM WebSocket (ws-client.ts)
  ├─ 消息去重（chatId+content 3秒窗口）
  ├─ 关键词检测 ['分析','需求澄清','技术可行性','出报告']
  ├─ DeepSeek 提取需求名称
  ├─ 飞书项目模糊搜索
  └─ 确认交互 / 直接触发
       │
       ▼
Express /trigger/:id (index.ts)
  │
  ▼
auto-analyzer.ts: handleNewRequirement()
  ├─ 运行中锁（同一 workItemId 不并发）
  ├─ 获取工作项详情（Meegle API）
  ├─ 提取 PRD（Wiki raw_content API）
  ├─ 缓存检查（14天，PRD 末变化则复用）
  ├─ 并行 AI 分析
  │   ├─ clarification.ts → 需求澄清
  │   └─ tech-feasibility.ts → 技术可行性
  ├─ createFormattedDoc → 上传 MD 到飞书云空间
  ├─ 写入缓存
  ├─ 授予文件权限（drive/v1/permissions）
  └─ 分发报告卡片
       ├─ 触发人 → chat_id 发送（IM 聊天窗口）
       ├─ 需求提出人 → open_id 发送（需求澄清）
       └─ 技术负责人 → open_id 发送（技术报告 + 需求澄清）
```

### 去重机制

| 层级 | 机制 | 说明 |
|------|------|------|
| WebSocket 事件 | `chatId:content` 3秒去重 | 防止飞书重复推送 |
| handleNewRequirement | `_running` Set 锁 | 同一 workItemId 不并发 |
| sendIM | `chatId:text` 2秒去重 | 文本消息防抖 |
| sendReportCard | `targetId:type:url` 5秒去重 | 卡片发送防抖 |
| 报告分发 | `servedClar` / `servedTech` 分别去重 | 同人不重发同类型报告 |

## 六、报告缓存

- 位置：`output/cache/` 目录
- 有效期：14 天
- 命中条件：`workItemId` 相同且 PRD 内容未变化
- 缓存命中时：跳过 AI 分析，直接复用已有文档，重新分发

## 七、与 claude-feishu 共存

如果同一台机器上也运行 `claude-feishu`（Claude Code 的飞书桥接）：

```env
# 在 config.env 中禁用 智慧智能体 的 WebSocket
FEISHU_WS_ENABLED=false
```

此时消息由 `claude-feishu` 接收，Claude Code 处理。

> 两者用同一套 `FEISHU_APP_ID` 时，消息会同时到达两个 WebSocket 客户端。**不建议同时启用**——除非两个系统处理不同聊天，否则会出现重复回复。

## 八、故障排查

| 症状 | 检查 | 修复 |
|------|------|------|
| 机器人无响应 | 服务器是否在运行？ | `curl http://localhost:3456/health` |
| 触发后返回 AI 闲聊 | 需求名没提取成功 | 用完整需求名，如「分析 萝卜蹲」 |
| 未找到匹配需求 | 飞书项目中需求名不一致 | 先确认飞书项目中的需求全名 |
| PRD 读取失败 | Wiki Token 是否过期？ | `http://localhost:3456/auth/feishu-login` 重新授权 |
| 文档生成为空 | DeepSeek API 是否正常？ | 检查 `DEEPSEEK_API_KEY` 和余额 |
| 报告卡片收不到 | chat_id 是否正确？ | 检查服务器控制台日志 |
| 重复收到报告 | 是否同时启用了 claude-feishu？ | 设 `FEISHU_WS_ENABLED=false` |

## 九、更新日志

### 2026-07-07

- **去重重构**：WebSocket 事件改用内容 hash 去重（修复不同 message_id 导致重复处理）
- **报告分发**：按报告类型（澄清/技术）分开去重，避免同一人收不到完整报告
- **运行中锁**：`handleNewRequirement` 加 `_running` Set，防止同一需求并发分析
- **chatId 透传**：触发人的报告优先用 `chat_id` 发送到聊天窗口（比 `open_id` 更可靠）
- **触发兜底**：提取不到需求名时引导用户明确指定，不再滑到普通聊天
- **发送级防抖**：`sendIM` 和 `sendReportCard` 加时间窗口去重
