## 三、配置说明

所有配置集中在 `config.env` 文件中，格式为 `字段名=值`。

### 必填项（必须配置，否则服务无法启动）

| 字段名 | 含义 | 示例值 |
|--------|------|--------|
| `FEISHU_PROJECT_PLUGIN_ID` | 飞书项目插件 ID | `MII_xxxxxxxxxxxx` |
| `FEISHU_PROJECT_PLUGIN_SECRET` | 飞书项目插件密钥 | `0154xxxxxxxxxxxx` |
| `FEISHU_PROJECT_USER_KEY` | 你的飞书项目用户标识 | `7649567855117192178` |
| `FEISHU_APP_ID` | 飞书机器人 App ID | `cli_xxxxxxxxxxxxx` |
| `FEISHU_APP_SECRET` | 飞书机器人 App Secret | `lvnncc4xxxxxxxxxxxxx` |
| `DEEPSEEK_API_KEY` | DeepSeek 大模型 Key | `sk-d7xxxxxxxxxxxxxxxxxxxx` |

### 可选项（有默认值，可不改）

| 字段名 | 含义 | 默认值 |
|--------|------|--------|
| `PROJECT_SPACE_KEY` | 飞书项目空间标识 | `aniwonder` |
| `AI_MODEL` | 大模型类型 | `deepseek` |
| `DEEPSEEK_BASE_URL` | DeepSeek API 地址 | `https://api.deepseek.com` |
| `PRD_REVIEW_TIMEOUT_DAYS` | PRD 超时提醒天数 | `3` |
| `CHECK_CRON` | 定时检查表达式 | `0 9 * * *` |
| `FEISHU_WS_ENABLED` | 是否启用飞书 IM WebSocket | `true` |
| `MEM0_API_KEY` | mem0 记忆 API Key（可选，不配则智能体无跨会话记忆） | 空 |

---

### 关于 WebSocket 路由

智小协服务通过 WebSocket 长连接直接接收飞书消息。**程序独立部署时，禁止任何其他进程（如 claude-feishu）连接同一机器人**，否则飞书随机投递消息，导致服务收不到。

| 场景 | FEISHU_WS_ENABLED | 说明 |
|------|:---:|------|
| **服务器部署（生产）** | `true`（默认） | 服务独占机器人，正常收发 |
| **本地开发 + Claude Code** | `false` | 关掉服务收件，消息走 Claude Code |
| **本地测试服务独立运行** | `true` | 关闭 claude-feishu 后再启动 |

> ⚠️ 生产环境部署到服务器后，只有一个智小协进程，不存在冲突。此配置仅本地开发时使用。

## 四、各平台凭据获取方法

### 4.1 飞书项目插件凭据

```
1. 打开飞书项目网页版 → https://project.feishu.cn
2. 左下角头像 → 开放平台
3. 创建插件（或使用已有插件）
4. 记录 Plugin ID 和 Plugin Secret
5. 填入 config.env 的 FEISHU_PROJECT_PLUGIN_ID 和 FEISHU_PROJECT_PLUGIN_SECRET
```

### 4.2 飞书项目 User Key

```
1. 打开飞书项目网页版
2. 左下角点击你的头像
3. 在弹窗中双击头像区域
4. User Key 已自动复制到剪贴板
5. 填入 config.env 的 FEISHU_PROJECT_USER_KEY
```

### 4.3 飞书机器人应用凭据

```
1. 打开飞书开放平台 → https://open.feishu.cn
2. 创建自建应用（或使用已有应用"智小协"）
3. 在应用首页获取 App ID 和 App Secret
4. 填入 config.env 的 FEISHU_APP_ID 和 FEISHU_APP_SECRET

此应用需要开通以下权限：
  - im:message（发送消息）
  - docx:document（创建文档）
  - wiki:wiki（读取知识库）
  - contact:contact:readonly（查询用户）
  - drive:drive.metadata:readonly（文档授权）
```

### 4.4 DeepSeek API Key

```
1. 打开 DeepSeek 平台 → https://platform.deepseek.com
2. 注册/登录 → API Keys → 创建新 Key
3. 复制 Key（sk- 开头）
4. 填入 config.env 的 DEEPSEEK_API_KEY
```

### 4.5 mem0 API Key（可选）

mem0 提供智能体跨会话记忆能力——分析前搜索历史决策，分析后存储结论。不配不影响核心功能。

```
1. 打开 mem0 平台 → https://app.mem0.ai/dashboard/api-keys
2. 注册/登录 → API Keys → 创建新 Key
3. 复制 Key（m0- 开头）
4. 填入 config.env 的 MEM0_API_KEY 或在 Claude Code settings.json 的 env 中添加
```


### 5.3 OAuth 授权 Wiki 读取

首次使用需要授权智小协读取飞书 Wiki 文档：
1. 浏览器打开 `http://localhost:3456/auth/feishu-login`
2. 在飞书授权页面点击「授权」
3. 看到「授权成功」即可

授权后 access_token 有效期 2 小时，过期后系统自动用 refresh_token 续期，**14 天内无需重新授权**。启动日志显示 `✅ Wiki Token 有效，自动续期中` 即正常。

### 5.4 手动触发测试

```bash
# 替换为实际的需求 ID（可选带触发人和聊天 ID）
curl -X POST "http://localhost:3456/trigger/new-requirement/7037200783?requester=ou_xxx&chat_id=oc_xxx"
```

### 5.5 分析流程详解

```
飞书用户 "分析 萝卜蹲"
  → 飞书 IM WebSocket 接收
  → 关键词检测 ['分析','需求澄清','技术可行性','出报告']
  → DeepSeek 提取需求名称
  → 飞书项目模糊搜索
  → 精准匹配 → 直接触发 / 模糊匹配 → 请用户确认
  → POST /trigger/:id → handleNewRequirement()
      ├─ 运行中锁（同一 workItemId 不并发）
      ├─ 获取工作项详情（Meegle API）
      ├─ 提取 PRD（Wiki raw_content API）
      ├─ 缓存检查（14天，PRD 未变化则复用）
      ├─ 并行执行两条分析管线：
      │   需求分析管线      技术可行性初评管线
      │   wiki:read         wiki:read
      │   mem0:search       mem0:search       ← 搜索历史记忆
      │   ai:analyze        ai:analyze        ← DeepSeek 分析
      │   mem0:remember     mem0:remember     ← 存入记忆
      │   docx:create       docx:create       ← 生成飞书文档
      │   im:send           im:send           ← 通知
      └─ 分发报告卡片（触发人/提出人/技术负责人）
```

**去重机制**：5 层防护

| 层级 | 机制 |
|------|------|
| WebSocket 事件 | chatId + content 3 秒去重 |
| handleNewRequirement | `_running` Set 锁（同 workItemId 不并发） |
| sendIM | chatId + text 2 秒去重 |
| sendReportCard | target + type + url 5 秒去重 |
| 报告分发 | `servedClar` / `servedTech` 按报告类型分开去重 |

### 5.6 报告分发逻辑

分析完成后，报告按以下规则分发：

| 收件人 | 收到什么 | 发送方式 |
|--------|----------|----------|
| 触发人 | 需求澄清 + 技术报告 | `chat_id` 发到聊天窗口 |
| 需求提出人 | 需求澄清 | `open_id` 私发 |
| 技术可行性确认节点负责人 | 技术报告 + 需求澄清 | `open_id` 私发 |

**去重机制**（按报告类型分开去重，同一人不会收到重复的同类报告）：

```
servedClar（需求澄清已发）
servedTech（技术报告已发）
  → 触发人先发 → 加入两个 Set
  → 提出人/技术负责人再发 → Set 命中则跳过
```

### 5.7 报告缓存

- 位置：`output/cache/` 目录（不入 git）
- 有效期：14 天
- 命中条件：同 `workItemId` 且 PRD 内容未变化
- 缓存命中时：跳过 AI 分析，复用已有文档 URL，重新分发

### 5.8 关键日志

| 前缀 | 含义 |
|------|------|
| `[IM]` | 飞书消息接收 |
| `[Tools]` | 内置工具注册数量 |
| `[SkillLoader]` | 外部技能加载（mem0 等） |
| `[Pipeline]` / `[Orchestrator]` | 分析管线执行步骤 |
| `[AutoAnalyzer]` | 分析流程（开始/PRD/缓存/完成） |
| `[Cache]` | 缓存命中/写入 |
| `[sendReportCard]` | 报告卡片发送 |
| `[WikiToken]` | Wiki 授权状态 + 自动续期 |

## 六、新增智能体

### 6.1 智能体是什么

智能体（Agent）= 一个 SKILL.md 文件，挂载在飞书项目流程节点上。当工作项流转到该节点时自动执行分析。

每个智能体由**一个 SKILL.md 文件**定义，包含两部分：

| 部分 | 内容 | 格式 |
|------|------|------|
| YAML 头部 | 元信息：名称、节点、需要的工具等 | `---` 包裹的 key: value |
| Markdown 正文 | 系统提示词：定义身份、行为、输出格式 | 自由编写 |

### 6.2 手动创建 SKILL.md

在 `agents/` 下新建目录和文件：

```markdown
---
name: 硬件方案评审
id: hardware-reviewer
description: 读取硬件方案文档，从EMC、散热、结构三方面评审
node: 硬件方案设计
department: 硬件部
tools:
  - wiki:read
  - ai:analyze
  - docx:create
  - im:send
output_target:
  - 触发人
  - 节点负责人
timeout_ms: 300000
---

你是资深硬件工程师，负责评审硬件方案设计。
请从以下维度逐一评审：

## 评审维度
1. 电磁兼容（EMC）— 布局、接地、屏蔽、滤波
2. 散热设计 — 热源分析、散热路径、裕量评估
3. 结构合理性 — 尺寸约束、安装方式、可制造性

## 输出格式
1. 总体评价
2. 各维度详细评审（含风险等级）
3. 改进建议
4. 结论
```

然后在 `agents/_registry.json` 中注册：

```json
{
  "agents": [
    "需求分析/SKILL.md",
    "技术可行性初评/SKILL.md",
    "硬件方案评审/SKILL.md"
  ]
}
```

保存后自动热加载，无需重启。

### 6.3 可用的内置工具

| 工具 ID | 功能 | 说明 |
|---------|------|------|
| `wiki:read` | 读取飞书 Wiki PRD | 自动从工作项提取 |
| `ai:analyze` | 调用 DeepSeek 分析 | 自动加载 agent-persona.md 底座 |
| `docx:create` | 生成飞书文档 | MD 上传 + docx fallback |
| `im:send` | 发送飞书消息卡片 | 通知触发人/负责人 |
| `project:query` | 查询飞书项目 | 工作项/节点/用户 |
| `project:search` | 搜索飞书项目需求 | 模糊匹配 |

### 6.4 外部技能

外部技能放在 `skills/` 目录下，自动发现、即插即用：

```bash
# 安装技能
cd skills/
git clone https://github.com/xxx/drawio-generator.git

# fs.watch 自动检测，无需重启
```

当前已安装：**mem0** — 提供 `mem0:search`（分析前搜历史）和 `mem0:remember`（分析后存结论）。

智能体要使用外部技能，只需在 SKILL.md 的 `tools` 中声明工具 ID 即可。

### 6.5 全局底座提示词

`knowledge/agent-persona.md` 是所有智能体的**全局底座**——自动注入到每次 AI 分析的系统提示词最前面。包含：

- 核心人设定位（具身智能专家 + 产品经理）
- 固定输出规则（去玄学、重落地、双视角输出）
- 第一红线（禁止改文档、禁止改代码）
- 触发规则、禁止行为

编辑此文件一次，全部智能体同步更新，无需逐个修改 SKILL.md。

### 6.6 架构说明

```
webhook/手动触发
  → Dispatcher 按节点匹配 Agent
  → agentExecutor 加载 SKILL.md
  → 标准分析管线（6 步）:
      wiki:read → mem0:search → ai:analyze → mem0:remember → docx:create → im:send
  → 每个 Agent 独立执行，互不影响
```

## 八、常见问题

**Q: 启动报错 "缺少必需配置"？**
A: 检查 `config.env` 文件是否存在，必填项是否都已填写。

**Q: 健康检查显示连接失败？**
A: 检查网络是否正常，飞书项目插件是否已安装到目标空间。

**Q: Wiki 读取失败 / 频繁要求重新授权？**
A: access_token 有效期 2 小时，系统会自动用 refresh_token 续期（14 天有效）。启动日志应显示 `✅ Wiki Token 有效，自动续期中`。如果频繁过期，检查 OAuth 授权是否正确完成。

**Q: DeepSeek 分析失败？**
A: 检查 `DEEPSEEK_API_KEY` 是否正确，Key 余额是否充足。

**Q: mem0 记忆功能不生效？**
A: 确认 `MEM0_API_KEY` 已配置，启动日志显示 `[SkillLoader] + mem0:search` 即正常。

**Q: 如何新增智能体？**
A: 手动在 `agents/` 下创建 SKILL.md 并注册到 `_registry.json`，保存后自动热加载。详见第六节。

**Q: 如何给智能体加新能力（如画图）？**
A: 把技能目录放到 `skills/` 下，`fs.watch` 自动检测加载。然后在智能体 SKILL.md 的 `tools` 中声明工具 ID 即可。

**Q: 如何定制智能体的全局行为？**
A: 编辑 `knowledge/agent-persona.md`，所有智能体自动继承。

**Q: 触发分析后收到 AI 闲聊，没有真正触发？**
A: 需求名称没有被正确提取。系统会引导重新输入。确保用「分析 + 需求名」格式。

**Q: 报告卡片重复收到？**
A: 已通过 5 层去重机制修复。如仍出现，检查是否同时运行了 claude-feishu（设 `FEISHU_WS_ENABLED=false`）。

**Q: 端口被占用？**
A: `PORT=3000 npm run dev` 换端口。
