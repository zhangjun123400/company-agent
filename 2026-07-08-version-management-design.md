# 版本管理智能体方案

> 日期: 2026-07-09 | 版本: v2.1 | 状态: 待审批

---

## 1. 背景

飞书项目已录入：

- **研发任务管理**（SRD，key: `6a43902cd74ffc7dc45d66e1`）：PRD 拆解后的开发任务，支持任意层级（通过 `父级任务` 字段关联，通常到三级）

**SRD 工作流规律**：

| 层级 | 工作流 | 特点 |
|------|--------|------|
| 父任务（有子任务） | 待开发 → 联调中 → 已完成 | 管理型流程，无模块节点 |
| 叶子任务（无子任务） | 待开发 → [模块并行] → 联调中 → 已完成 | 开发型流程，模块节点并行 |

- 模块节点（运控、软件中台、感知、导航、嵌软、客户端、云端等）**并行执行**（相同 `actual_begin_time`）
- 父任务的 `是否完成` 字段**不可信**（存在父任务标记完成但子任务未完成的不一致情况），进度计算必须**递归检查所有叶子节点**
- **版本开发**（key: `6a41e22f1dcaa1da30c0ca94`）：组合 PRD + SRD 的发布版本，7 个节点工作流（版本需求整理→开发→联调→提测→测试→门禁评审→上线），关键字段 `本版PRD`、`跟版SRD`、`SRD开发完成进度`

当前系统已覆盖需求分析和节点超时提醒，版本维度的管理空白。

## 2. 能力 1：版本启动后人力盘点

### 2.1 触发时机

两种触发方式：

- **自动**：任一版本「版本需求整理」节点完成时触发
- **手动**：飞书发「版本盘点」或「人力盘点」

### 2.2 盘点维度

| 维度      | 数据来源                | 说明                |
| ------- | ------------------- | ----------------- |
| 版本全景    | 版本开发工作项             | 当前几个版本在跑、各处于什么阶段  |
| 人力分布    | SRD `所属模块` + `创建者`  | 按模块统计人员、任务数、人均负载  |
| 人员复用率   | 跨版本交叉分析             | 同一人出现在几个版本中       |
| 版本节奏    | 版本实际 vs 计划对比        | 历史版本节奏数据          |
| 角色覆盖    | SRD `role_assignee` | PM/SE/Dev/QA 是否到位 |
| 门禁通过率   | 版本 `门禁评审` 节点        | 历史门禁通过/驳回统计       |
| NUDD 风险 | 风险管理 `归属版本` 字段      | 关联风险数量、等级分布、处理进度  |

### 3.4 节点时间来源

飞书项目 API 返回的每个节点有 `actual_begin_time` 和 `actual_finish_time`。已完成节点用实际时间，待开始节点的时间从 `config/reminder-rules.json` 的节点周期配置中取：

```json
{
  "nodeDurations": {
    "开发": 10,
    "联调": 4,
    "测试": 6,
    "门禁评审": 1
  }
}
```

以上一个已完成节点的 `actual_finish_time` 为起点，逐节点累加时长，估算后续节点日期。

## 4. 能力 3：进度偏离提醒

### 4.1 判定标准

| 概念       | 公式                                             |
| -------- | ---------------------------------------------- |
| **预期进度** | (当前时间 − 开发节点开始时间) ÷ (开发节点计划时长) × 100%（上限 100%） |
| **实际进度** | 递归统计所有叶子 SRD：已完成叶子数 ÷ 总叶子数 × 100%（**不用** `SRD开发完成进度` 字段，递归计算更准确） |
| **偏离**   | 实际进度 − 预期进度                                    |

```
偏离 ≥ 0  → 正常或超前，不提醒
偏离 ≤ −15% → 落后超过 15%，触发提醒
```

**举例**：开发节点计划 10 天，已过 6.5 天 → 预期 65%。版本下 2 个一级任务共 5 个叶子 SRD，递归统计：3 个叶子的「已完成」节点 status=3 → 实际 60%。偏离 = 60% − 65% = −5% → 正常。

> 若用不可信的 `是否完成` 字段：父任务标记完成 → 算 2/2 = 100%，虚高，掩盖了 2 个叶子未完成的真实情况。

### 4.2 节点时长配置

每个节点的计划时长在 `config/reminder-rules.json` 中配置：

```json
{
  "id": "version-progress-deviation",
  "name": "版本进度偏离提醒",
  "trigger": {
    "on": "version_progress",
    "workItemType": "版本开发",
    "deviationThreshold": 15
  },
  "targetNode": "开发",
  "nodeDurations": {
    "开发": 10,
    "联调": 4,
    "提测": 4,
    "测试": 6,
    "门禁评审": 1,
    "上线": 1
  },
  "remind": {
    "target": "角色",
    "roles": ["PM", "SE"]
  },
  "enabled": true
}
```

### 4.4 只检查当前活跃节点

不检查所有节点——只检查当前 `status=2`（进行中）的节点。版本可能同时只有 1 个节点在进行中，按该节点的计划时长计算预期进度。

### 5.4 进度计算核心逻辑

```typescript
/** 递归收集所有叶子 SRD（无子任务的最底层节点） */
function collectLeaves(srdIds: number[], allSRDs: Map<number, SRD>): SRD[] {
  const leaves: SRD[] = [];
  for (const id of srdIds) {
    const srd = allSRDs.get(id);
    if (!srd) continue;
    const children = allSRDs.filter(s => s.parentId === id);
    if (children.length === 0) {
      leaves.push(srd);          // 叶子节点
    } else {
      leaves.push(...collectLeaves(children.map(c => c.id), allSRDs)); // 递归
    }
  }
  return leaves;
}

function calcDeviation(version, allSRDs, nodeDurations) {
  const activeNode = version.workflow_nodes.find(n => n.status === 2);
  if (!activeNode) return null;

  const plannedDays = nodeDurations[activeNode.name];
  if (!plannedDays) return null;

  // 预期进度：按时间线性
  const elapsed = (Date.now() - new Date(activeNode.actual_begin_time).getTime()) / (24*3600*1000);
  const expected = Math.min(100, (elapsed / plannedDays) * 100);

  // 实际进度：递归统计叶子 SRD 的「已完成」节点
  const srdIds = version.fields.跟版SRD;
  const leaves = collectLeaves(srdIds, allSRDs);
  const total = leaves.length;
  const completed = leaves.filter(s =>
    s.workflow_nodes?.find(n => n.name === '已完成')?.status === 3
  ).length;
  const actual = total > 0 ? (completed / total) * 100 : 0;

  return {
    expected: Math.round(expected),
    actual: Math.round(actual),
    deviation: Math.round(actual - expected),
  };
}
```
